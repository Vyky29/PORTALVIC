How roster changes flow in Admin

This guide is for anyone using the Admin dashboard. It explains where each screen writes data, what staff see on their tablets, and which tool to pick so Admin, Staff, and session feedback stay aligned.

1. Three layers (not one spreadsheet)

Staff and Lead dashboards merge three sources. Admin tools usually touch layers 2 or 3. The whole-term baseline (layer 1) is shipped with each deploy.

Layer 1

Term bundle (deployed baseline)

staff_dashboard_spreadsheet_bundle.js — default grid for the term: participant, time, venue, pool/area, instructor. Built from MADRE (roster_term_master.json) by the technical owner. Updates after git push → Vercel (~5–10 min).

Layer 2

Live term roster (Supabase)

Table portal_roster_rows — who is booked in a slot for the rest of term (or selected dates). Written by New participant and Edit term slot. Visible to staff after refresh / realtime merge — no redeploy.

Layer 3

One-day overrides (Supabase)

Table schedule_overrides — absent, cancel, cover, make-up, move for one calendar day only. Written by Schedule & Covers. Also live — no redeploy.

2. End-to-end flow (what happens when you save)

Pick the right admin screen → data lands in the right store → merged view updates for staff.

4. Decision tree (30 seconds)

Golden rule: one logical change → one tool. Term-wide participant changes go through Edit term slot (layer 2). Single-day exceptions go through Schedule & Covers (layer 3). The deployed bundle (layer 1) is the default skeleton underneath both.

5. Before you save — quick checklist

  1. Is this one day or every week until end of term? Pick the matching tool and scope.
  2. Participant name, date anchor, time band, service, pool/area, and instructor all correct?
  3. After save: open Staff dashboard for that instructor on the same date and confirm the day card.
  4. If you changed a weekday pattern for the rest of term, tell the technical owner to fold it into MADRE when convenient (keeps layer 1 and 2 from drifting).
  5. Session feedback counts follow merge rules (e.g. Aquatic + Multi-Activity same instructor = one form on the tablet, separate counters in “this week”).

6. If Admin and Staff disagree

SymptomLikely causeWhat to do
Staff still show old name after term swap Only a one-day override saved, or bundle still has old name Use Edit term slot with weekday scope; avoid cancelling old name in Schedule & Covers
Red banner in Schedule & Covers Live roster (layer 2) ≠ bundle name (layer 1) Click Edit term slot from the drawer — do not fight layers
New trial not on tablet Save failed (auth) or wrong open cell Re-save New participant; staff refresh portal; check Supabase if needed
Whole term baseline wrong everywhere Layer 1 out of date Request MADRE sync + deploy from technical owner
